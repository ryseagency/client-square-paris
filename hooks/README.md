# RYSE — Hooks Claude Code

3 hooks qui enforcent les règles RYSE au niveau **système** (100% au lieu des 70% via instructions CLAUDE.md seules).

## Ce que font les hooks

### 1. `pre-edit-protect.sh` — PreToolUse

Bloque (exit 2) toute édition contenant des patterns interdits dans le **code livrable** :

- Commentaires IA / origine (`generated by Claude`, `// AI`, `{% comment %} TODO Claude %}`)
- TODO / FIXME / XXX / HACK
- `console.log` / `debugger` / `print()` de debug
- Variables qui leakent l'origine (`aiGenerated`, `lovableTmp`, `_v2_final`)
- `"default": ""` sur `text/textarea/richtext/inline_richtext` (erreur Shopify)

**Skip automatique** sur les fichiers non-livrables : `tasks/`, `lessons.md`, `scripts/`, `.claude/`, `README*`, `docs/`, `CLAUDE.md`, `CLIENT.md`.

### 2. `post-edit-quality.sh` — PostToolUse

Après chaque édition d'un `.liquid` : lance `shopify theme check --severity error` ciblé. Si erreurs → message à Claude pour qu'il corrige au prochain tour.

Bonus : valide la syntaxe JSON sur édition de `*.json`.

### 3. `stop-verification.sh` — Stop

Avant que Claude rende la main : Theme Check global. Si erreurs → bloque (`decision: block`) avec rappel de la règle Boris Cherny : *« never mark a task complete without proving it works ».*

## Installation

Depuis ce dossier :

```bash
./install-hooks.sh /chemin/vers/projet/client
```

Ou depuis le projet client :

```bash
~/Documents/Claude/RYSE/_templates/hooks/install-hooks.sh .
```

Le script :
- Crée `.claude/hooks/` dans le projet
- Copie les 3 scripts + les rend executable
- Copie `settings.json` (ou alerte s'il existe déjà — à merger manuellement)
- Vérifie les dépendances (`jq`, `shopify` CLI)

## Dépendances

- `jq` (parser JSON sur stdin) — `brew install jq`
- `shopify` CLI (Theme Check) — `brew tap shopify/shopify && brew install shopify-cli`

## Désactivation temporaire

Si un hook bloque une action légitime (ex : édition d'un script de test contenant volontairement `console.log`) :

```bash
mv .claude/settings.json .claude/settings.json.off
# … faire l'action …
mv .claude/settings.json.off .claude/settings.json
```

Si la règle est inadaptée à un contexte récurrent : ajuster le hook (ajouter une exception path), pas le contourner.

## Debug

Voir ce qu'un hook a renvoyé à Claude : la stderr est visible dans la conversation Claude Code juste après le tool call.

Tester un hook manuellement :

```bash
# Tester pre-edit-protect avec un payload simulé
echo '{"tool_input":{"file_path":"sections/hero.liquid","new_str":"{% comment %} TODO Claude {% endcomment %}"}}' | .claude/hooks/pre-edit-protect.sh
echo "Exit: $?"
# Devrait retourner exit 2 + message d'erreur
```

## Exit codes Claude Code

- `0` = OK, action autorisée (ou simple log informatif côté PostToolUse)
- `2` = BLOCK (PreToolUse) ou FORCE CONTINUE (Stop) — message stderr lu par Claude
- autre non-zero = erreur non-bloquante, message visible à l'utilisateur seulement

## Maintenance

Quand on identifie une nouvelle catégorie d'erreur récurrente :
1. Ajouter d'abord la règle dans `lessons.md` du projet (ou `CLAUDE.md` global si universel)
2. Si la règle est mécaniquement vérifiable → l'ajouter au hook concerné
3. Tester avec un payload simulé avant commit
